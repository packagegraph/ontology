# CQ Evidence Manifest — Generation Specification

**Purpose:** Define the output shape and rubric for a generated artifact that maps each CQ to its required properties, join sides, and minimum evidence for PASS. This manifest is generated from the frozen CQ file by the validation harness — NOT hand-maintained.

**Owner:** Ontology team defines the rubric. Platform validation harness generates the artifact.

---

## Output Shape

The manifest is a JSON file generated by parsing each CQ's SPARQL query. For each CQ:

```json
{
  "cq_id": "CQ-SCR-06",
  "title": "Patch Lag by Distribution — Comparative Time-to-Fix",
  "declared_status": "ADVISORY-SIDE SATISFIED",
  "required_properties": [
    {"property": "sec:publishedDate", "side": "vulnerability", "domain": "Vulnerability"},
    {"property": "sec:addressesVulnerability", "side": "advisory", "domain": "SecurityAdvisory"},
    {"property": "sec:advisoryId", "side": "advisory", "domain": "SecurityAdvisory"},
    {"property": "sec:advisoryDate", "side": "advisory", "domain": "SecurityAdvisory"},
    {"property": "sec:advisoryForPackage", "side": "advisory", "domain": "SecurityAdvisory"},
    {"property": "pkg:partOfRelease", "side": "package", "domain": "Package"}
  ],
  "join_type": "two-sided",
  "join_sides": {
    "advisory": ["sec:advisoryForPackage", "sec:advisoryDate", "sec:addressesVulnerability"],
    "vulnerability": ["sec:publishedDate"]
  },
  "pass_criteria": "Both advisory-side and vulnerability-side properties have >0 triples in the target graph, AND the end-to-end query returns >=5 rows (MARGINAL threshold for analytical CQs)",
  "partial_criteria": {
    "ADVISORY-SIDE SATISFIED": "All advisory-side properties have >0 triples",
    "VULNERABILITY-SIDE SATISFIED": "All vulnerability-side properties have >0 triples"
  }
}
```

## Property Classification Rules

The harness classifies each property extracted from the SPARQL by namespace:

| Namespace Prefix | Side | Examples |
|-----------------|------|----------|
| `sec:advisory*`, `sec:addressesVulnerability`, `sec:advisoryForPackage` | advisory | advisoryDate, advisoryId, advisorySeverity |
| `sec:published*`, `sec:hasCVSSScore`, `sec:baseScore`, `sec:hasAffectedRange`, `sec:affects*`, `sec:cveId`, `sec:osvId` | vulnerability | publishedDate, hasCVSSScore, affectsEcosystem |
| `pkg:*` | package | partOfRelease, packageName, maintainedBy |
| `prov:*`, `foaf:*`, `rdfs:*` | infrastructure | wasAttributedTo, name, label |

## Join Type Classification

| Condition | Join Type |
|-----------|-----------|
| CQ uses ONLY vulnerability-side `sec:` properties | `vulnerability-only` |
| CQ uses ONLY advisory-side `sec:` properties | `advisory-only` |
| CQ uses BOTH advisory and vulnerability `sec:` properties | `two-sided` |
| CQ uses only `pkg:` properties | `package-only` |

## PASS Criteria

A CQ is PASS when:

1. ALL required properties have >0 triples in the target Fuseki dataset
2. The end-to-end SPARQL query returns >=5 rows (MARGINAL threshold)
3. For two-sided CQs: both sides must be independently satisfied

A CQ is MARGINAL when:

1. The query returns 1-4 rows
2. Evidence suggests the rows come from example/test fixtures rather than production data

## Generation Contract

The validation harness (`platform/etl/scripts/cq-validate.py`) should:

1. Parse SPARQL from the frozen CQ file
2. Extract all property URIs from triple patterns
3. Classify each property by side (advisory/vulnerability/package)
4. Run property-count probes: `SELECT (COUNT(*) AS ?c) WHERE { ?s <property> ?o }` for each
5. Emit the manifest JSON with per-CQ entries
6. Cross-reference the manifest against query results to produce final status

## Rubric for Platform Reports

When the platform team reports CQ impact from a new collector or enricher:

1. **Check the manifest** — which CQs list the new properties?
2. **Check the side** — does the new data satisfy the advisory side, vulnerability side, or both?
3. **Report the correct status** — use the Status Vocabulary from `competency-questions.md`
4. **Do not claim PASS** without end-to-end query evidence (>=5 rows)
